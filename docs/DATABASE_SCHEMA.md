# Onboarding Buddy Database Schema

## 1. Database Goal

The database is responsible for storing all structured onboarding workflow data required for:

* workflow continuity
* onboarding task management
* approval persistence
* audit logging
* retry handling
* workflow observability
* debugging
* workflow recovery
* agent execution history
* supervisor routing visibility
* workflow state persistence

The database acts as the persistent memory layer of the supervisor-based multi-agent onboarding system.

---

# 2. Database Philosophy

The system intentionally avoids relying entirely on:

* LLM memory
* chat history
* prompt continuation
* temporary workflow state stored only in RAM

Instead, all critical workflow data should remain:

* persistent
* structured
* queryable
* observable
* recoverable
* traceable

The database is considered a core workflow infrastructure component.

The database is intentionally designed to support:

```text id="rjlwmx"
Supervisor Agent
+
Specialist Agents
+
Workflow State Persistence
+
Observability
```

# 14. Workflow Persistence Strategy

The workflow state should persist across:

* retries
* approval waiting
* workflow failures
* backend restarts
* temporary infrastructure failures

Important workflow state must never exist only in RAM.

Critical state should remain persisted inside:

* workflow_runs
* workflow_state
* audit_logs
* agent_runs
* onboarding_tasks

The Supervisor Agent relies on persisted workflow state for reliable routing decisions.

---

# 15. Retry Persistence

Retry information should remain stored persistently.

Retry persistence improves:

* workflow recovery
* debugging
* observability
* retry safety
* failure analysis

Retry tracking fields:

* retry_count
* failure_reason
* current_node
* workflow_status
* execution_status

Retries should remain observable across:

* workflow level
* specialist agent level
* tool execution level

---

# 16. Observability Persistence

Observability data should remain queryable.

The system should preserve:

* workflow transitions
* approval decisions
* retry attempts
* failures
* workflow completion history
* supervisor routing decisions
* specialist agent execution history
* routing reasons
* execution durations

Observability data is considered production-critical infrastructure.

---

# 17. Workflow State Recovery Strategy

The persisted workflow state allows safe recovery after interruption.

Recovery flow:

```text id="7mk40x"
Workflow Failure
↓
Load Persisted Workflow State
↓
Restore Current Node
↓
Restore Current Agent
↓
Restore Retry Metadata
↓
Resume Safe Workflow Execution
```

The system should never lose onboarding workflow state silently.

---

# 18. Database Indexing Strategy

The MVP indexing strategy should prioritize:

* workflow lookup speed
* employee lookup speed
* audit log retrieval
* agent execution tracing

Recommended indexes:

| Table            | Indexed Fields               |
| ---------------- | ---------------------------- |
| employees        | employee_email               |
| onboarding_tasks | employee_id, task_status     |
| approvals        | employee_id, approval_status |
| audit_logs       | employee_id, workflow_run_id |
| workflow_runs    | employee_id, workflow_status |
| agent_runs       | workflow_run_id, agent_name  |
| workflow_state   | workflow_run_id              |

SQLite indexing should remain lightweight during MVP development.
